# day1总结

```go
package main // 声明这个文件属于 main 包，这是程序的入口包

import "fmt" // 导入标准库 fmt，相当于 import { ... } from 'fmt'

func main() { // 这是程序的入口函数，程序从这里开始运行
    fmt.Println("Hello Go") // 在控制台打印字符串
}
```

既然你是前端开发者，咱们把这段代码拆解开，通过**“类比”**的方式来看，你会发现它的结构逻辑非常清晰：

### 1. `package main` —— “定义模块作用域”
*   **代码作用：** 声明当前文件所属的“包”（Package）。
*   **类比理解：** 这就像是 JS 里的 `export` 或者模块的命名空间。
*   **深层含义：** 在 Go 中，`main` 是一个**特殊保留字**。当你写了 `package main`，Go 编译器就会知道：“嘿，这个项目编译完后是要作为一个**可执行程序**（二进制文件）运行的，而不是一个库（Library）。” 如果你写的是 `package utils`，它会被编译成一个包，只能被别人 `import` 使用。

### 2. `import "fmt"` —— “加载依赖”
*   **代码作用：** 引入 Go 的标准库 `fmt`（Format 的缩写）。
*   **类比理解：** 这相当于 ES6 的 `import fmt from 'fmt'`。
*   **深层含义：** `fmt` 是 Go 最常用的标准库之一，专门负责**格式化 I/O**（输入输出），比如在控制台打印日志、格式化字符串等。Go 语言非常严谨，**引入了包却不使用，编译会直接报错**（不像 JS 可能会报 Warning），这强制保证了代码的整洁。

### 3. `func main() { ... }` —— “程序的主入口”
*   **代码作用：** 定义一个名为 `main` 的函数，它是程序的**启动入口点**。
*   **类比理解：** 想象一下浏览器执行 JS 时，虽然没有显式的 `main` 函数，但它默认是从文件第一行开始自上而下执行的。在 Go 中，程序的执行逻辑是强制从 `main` 包下的 `main` 函数开始的。
*   **深层含义：** 函数体用 `{}` 包裹，这是 Go 的语法规则。注意，这里的 `{` **必须跟在函数名后面的一行开头**，如果你换行写 `{`，编译器会报错（这是 Go 特有的“强制格式化”逻辑）。

### 4. `fmt.Println("Hello Go")` —— “具体执行动作”
*   **代码作用：** 调用 `fmt` 包中的 `Println` 函数，将字符串输出到标准输出（控制台）并换行。
*   **类比理解：** 这就是 JS 里的 `console.log("Hello Go")`。
*   **深层含义：** 这里的 `.` 操作符表示调用某个包里的公开方法（首字母大写在 Go 中意味着该方法是对外导出的 Public 方法）。

---

### 总结：这段代码的“生命周期”
当你运行 `go run main.go` 时，Go 编译器做了三件事：
1. **寻找 `main` 包**：确认这是个可执行程序。
2. **连接依赖**：找到并加载 `fmt` 这个标准库。
3. **寻找 `main` 函数**：将控制权交给 `main` 函数，执行里面的代码，最后结束程序。

**给你的启发：**
在 JS 中，我们习惯了“弱类型”和“自由度”，但在 Go 里，程序入口必须唯一（`main`）、包必须显式声明、代码必须严格格式化。**虽然起步时觉得条条框框多，但到了大型复杂项目时，你会发现这种“高度约束”会让代码的可读性和维护性比 JS 高出好几个档次。**
